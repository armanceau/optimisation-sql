## Exercice 1 : Analyse d'une requête simple

- Quel est le temps d'exécution total ? ...

**Répondez aux questions :**

1. Pourquoi PostgreSQL utilise-t-il un Parallel Sequential Scan ? : Car il y'a de nombreuses lignes et qu'il n'y a pas d'index
2. La parallélisation est-elle justifiée ici ? Pourquoi ? Oui car beaucoup de ligne doivent être lues et testées
3. Que représente la valeur "Rows Removed by Filter" ? Le nombrer de ligne lues et rejetées car elles ne correspondent pas à la condition

### 1.3 Création d'index

Créez un index simple sur la colonne `start_year` de la table `title_basics`.

### 1.4 Analyse après indexation

Exécutez à nouveau la même requête que dans l'étape 1.1 et comparez les résultats.

### 1.5 Impact du nombre de colonnes

Modifiez votre requête pour sélectionner uniquement les colonnes `tconst`, `primary_title` et `start_year` (au lieu de toutes les colonnes avec `*`).

Exécutez cette nouvelle requête avec `EXPLAIN ANALYZE` et comparez les résultats avec ceux de l'étape 1.4.

```sql
EXPLAIN ANALYZE SELECT tconst, primary_Title, start_Year FROM title_basics WHERE start_year = 2020;
```

1. Le temps d'exécution a-t-il changé ? Pourquoi ? Oui il est plus rapide, car il y'a moins de données à lire par ligne
2. Le plan d'exécution est-il différent ? oui, il peut passer sur un Index Only Scan
3. Pourquoi la sélection de moins de colonnes peut-elle améliorer les performances ? Car il y'a moins de donneés à lire et à trasnférer

### 1.6 Analyse de l'impact global

Comparez avec le plan de l'étape 1.1 :

1. Quelle nouvelle stratégie PostgreSQL utilise-t-il maintenant ? Bitmap Index Scan
2. Le temps d'exécution s'est-il amélioré ? De combien ? Oui, de quelques secondes
3. Que signifie "Bitmap Heap Scan" et "Bitmap Index Scan" ?

4. Pourquoi l'amélioration n'est-elle pas plus importante ? Parce que même avec l'index, PostreSQL doit encore accéder à la table pour récupérer les colonnees non couvertes par l'index

## Exercice 2 : Requêtes avec filtres multiples

### 2.1 Requête avec conditions multiples

Écrivez une requête avec `EXPLAIN ANALYZE` qui recherche tous les films (`title_type = 'movie'`) sortis en 1950.

```sql
EXPLAIN ANALYZE SELECT * FROM title_basics WHERE start_year = 1950 AND title_type = 'movie';
```

### 2.2 Analyse du plan d'exécution

1. Quelle stratégie est utilisée pour le filtre sur `start_year` ? Bitmap Heap Scan
2. Comment est traité le filtre sur `title_type` ? Le filtre est apppliqué après coup en mémoire
3. Combien de lignes passent le premier filtre, puis le second ? 8707
4. Quelles sont les limitations de notre index actuel ? 2007

### 2.3 Index composite

Créez un index composite qui combine les colonnes `start_year` et `title_type`.

```sql
CREATE INDEX idx_title_basics_year_type ON title_basics (start_year, title_type);
```

### 2.4 Analyse après index composite

Exécutez à nouveau la même requête qu'à l'étape 2.1 et observez les différences. La requête a été très rapide, on obtient 2007 lignes directement, sans aucune ligne rejetée par un filtre après coup (le Rows Removed by Filter a disparu). Le nombre de blocs lus (Heap Blocks) est passé de 3525 à 944 grâce à une bitmap plus précise, ce qui explique la baisse du temps d'exécution (8.6 ms → 4.1 ms).

### 2.5 Impact du nombre de colonnes

Modifiez votre requête pour sélectionner uniquement les colonnes `tconst`, `primary_title`, `start_year` et `title_type`.

```sql
EXPLAIN ANALYZE
SELECT tconst, primary_title, start_year, title_type
FROM title_basics
WHERE start_year = 1950 AND title_type = 'movie';
```

Comparez les performances avec la requête de l'étape 2.4 :

1. Le temps d'exécution a-t-il changé ? Oui, légèrement plus rapide, car moins de données sont transférées
2. Pourquoi cette optimisation est-elle plus ou moins efficace que dans l'exercice 1 ? moins efficace, car le gain vient essentiellement déjà de l'index
3. Dans quel cas un "covering index" (index qui contient toutes les colonnes de la requête) serait idéal ? Si l'index contenait aussi `tconst` et `primary_title` PostgreSQL pourrait répondre entièrement depuis l'index sans aucun accès à la table élmininant complètement les heap blocs

### 2.6 Analyse de l'amélioration globale

1. Quelle est la différence de temps d'exécution par rapport à l'étape 2.1 ? Passage de ~8.6 ms à ~4 ms
2. Comment l'index composite modifie-t-il la stratégie ? le filtre `title_type` auparavant appliqué après coup fait maintenant partie de l'index, les 2 sont évalués directment dans via l'index
3. Pourquoi le nombre de blocs lus ("Heap Blocks") a-t-il diminué ? parce que le bitmap généré est plus précis
4. Dans quels cas un index composite est-il particulièrement efficace ? Quand les requêtes filtrent régulièrement sur la même combinaison de colonnes ensemble, surtout si la première colonne de l'index est très sélective et que la seconde condition élimine encore beaucoup de faux positifs.

## Exercice 3 : Jointures et filtres

### 3.1 Jointure avec filtres

Écrivez une requête avec `EXPLAIN ANALYZE` qui joint les tables `title_basics` et `title_ratings` pour trouver le titre et la note des films sortis en 1994 ayant une note moyenne supérieure à 8.5.

```sql
EXPLAIN ANALYZE
SELECT tb.primary_title, tr.average_rating
FROM title_basics tb
JOIN title_ratings tr ON tb.tconst = tr.tconst
WHERE tb.start_year = 1994 AND tr.average_rating > 8.5;
```

### 3.2 Analyse du plan de jointure

1. Quel algorithme de jointure est utilisé ? Nested Loop : boucle sur les lignes de `title_ratings filtrées`, et pour chacune cherche la ligne correspondante dans `title_basics`
2. Comment l'index sur `start_year` est-il utilisé ? il n'est pas utilisé
3. Comment est traitée la condition sur `average_rating` ? Par un Filter: `average_rating > 8.5` appliqué lors d'un Seq Scan complet sur `title_ratings`
4. Pourquoi PostgreSQL utilise-t-il le parallélisme ? Il ne l'utilise pas, il considère qu'il y'a trop peu de ligne

### 3.3 Indexation de la seconde condition

Créez un index sur la colonne `average_rating` de la table `title_ratings`.

```sql
CREATE INDEX idx_title_rating_average ON title_ratings (average_rating);
```

### 3.4 Analyse après indexation

Exécutez à nouveau la requête de l'étape 3.1 et observez les différences.

Le Seq Scan sur title_ratings est remplacé par un Bitmap Heap Scan + Bitmap Index Scan utilisant idx_title_rating_average. Le filtre average_rating > 8.5 est maintenant appliqué via l'index (Index Cond) plutôt que testé ligne par ligne sur toute la table. Le temps d'exécution passe de 0.247 ms à 0.038 ms.

### 3.5 Analyse de l'impact

1. L'algorithme de jointure a-t-il changé ? Non c'est toujours Nested Loop
2. Comment l'index sur `average_rating` est-il utilisé ? Via un Bitmap Index Scan sur idx_title_rating_average avec Index Cond: average_rating > 8.5

## Exercice 6 : Synthèse et réflexion

### 1. Quand un index est-il le plus efficace ?

- **Recherches ponctuelles ou volumineuses ?**
  Pour les recherches ponctuelles (peu de lignes retournées par rapport au total). Si une requête retourne une grande partie de la table, un Seq Scan reste souvent plus rapide qu'un index (moins d'accès disque aléatoires).

- **Colonnes avec forte ou faible cardinalité ?**
  Forte cardinalité (beaucoup de valeurs distinctes, ex: `tconst`). Sur une colonne à faible cardinalité (ex: un booléen), l'index filtre peu de lignes et PostgreSQL préfère souvent un Seq Scan.

- **Opérations de type égalité ou intervalle ?**
  Les deux fonctionnent bien avec un index B-tree (par défaut), mais l'égalité (`=`) est généralement la plus efficace ; les intervalles (`>`, `<`, `BETWEEN`) restent performants tant que la plage sélectionnée reste petite par rapport à la table.

### 2. Quels algorithmes de jointure avez-vous observés ?

- **Nested Loop** : observé dans l'exercice 3. Efficace quand une des deux tables (ou le résultat filtré) est petite, surtout combiné à un index sur la colonne de jointure côté table interne.

- **Dans quels contextes chacun est-il préféré ?**
  - Nested Loop : petit volume d'un côté, avec un bon index disponible pour l'autre côté.
  - Hash Join : gros volumes des deux côtés, pas d'index utile, la table la plus petite tient en mémoire pour construire la table de hachage.
  - Merge Join : les deux ensembles sont déjà triés (ou peuvent l'être facilement), utile pour de gros volumes avec un ordre exploitable.

- **Comment le volume de données influence-t-il le choix ?**
  Plus le volume de lignes à joindre est petit, plus Nested Loop devient avantageux. Quand les volumes augmentent des deux côtés, PostgreSQL bascule vers Hash Join ou Merge Join, plus efficaces à grande échelle malgré un coût de démarrage plus élevé.

### 3. Quand le parallélisme est-il activé ?

- Quand le coût estimé du travail à faire (scan, agrégation, tri) dépasse un seuil (table volumineuse, beaucoup de lignes à lire/traiter) où le gain de répartir le travail sur plusieurs workers dépasse le coût de coordination.

- **Quels types d'opérations en bénéficient le plus ?**
  Les Seq Scans sur de grosses tables, les agrégations (`Partial`/`Finalize Aggregate`), les tris sur de gros volumes.

- **Pourquoi n'est-il pas toujours utilisé ?**
  Parce que pour de petits volumes de données, l'overhead de lancement et de coordination des workers coûte plus cher que le gain — on l'a vu dans l'exercice 3, où les Nested Loop restent séquentiels car les volumes filtrés sont trop faibles.

### 4. Quels types d'index utiliser dans les cas suivants ?

- **Recherche exacte sur une colonne** → Index B-tree simple.
- **Filtrage sur plusieurs colonnes combinées** → Index composite (multicolonne), avec la colonne la plus sélective/la plus souvent utilisée seule en premier.
- **Tri fréquent sur une colonne** → Index B-tree sur cette colonne (l'ordre de l'index évite un tri explicite en mémoire).
- **Jointures fréquentes entre tables** → Index sur la ou les colonnes de jointure (souvent déjà couvert si c'est une clé primaire/étrangère, sinon créer un index dédié).
